# 编码风格

## 适用场景

在新增或修改后端业务代码、前端页面逻辑、公共组件、配置装配、复杂流程控制或兼容性分支时，读取本文件。

## 强制规则

- 编码风格优先跟随邻近模块和现有文件，不按个人偏好额外切换命名、缩进、空行、import 排序、引号或拆行习惯。
- “跟随邻近模块和现有文件”主要约束命名、缩进、空行、import 排序、引号、拆行和局部组织方式，不意味着可以继承历史遗留的缺注释状态。
- 只做与当前任务直接相关的格式调整；若没有明确收益，不顺手重排整段代码、整个文件或整批目录。
- 变量、方法、类、组件和 DTO 命名优先使用业务语义清晰的名称，避免只靠缩写、序号或临时语义命名。
- 代码注释统一使用中文；只有在保留协议字段、接口字段、第三方术语、类库关键字或错误原文时，才保留必要英文。
- 若目标文件当前几乎没有注释，这属于存量历史状态，不构成当前任务的豁免理由；本次新增或修改到的业务类、关键方法、复杂代码块，仍按本文件补齐必要中文注释。
- 新增或修改类、接口、枚举、实体、DTO、VO、Controller、Service、Extract、Job、配置类时，若文件承载明确业务职责，默认补齐中文类注释；不要出现“实现已改完但文件顶部没有任何中文职责说明”的情况。
- 新增或修改 public / protected 方法时，若方法涉及业务规则、跨插件调用、宿主约束、配置装配、权限判断、状态流转、兼容分支或副作用，默认补齐中文方法注释或紧贴代码块的中文说明；不要把阅读成本完全留给后续维护者自己推断。
- 注释优先解释业务意图、边界条件、时序约束、兼容原因和“为什么这样做”，不要重复代码表层动作。
- 注释多少按业务复杂度决定：简单直白的赋值、映射、样板接线代码不强行加注释；涉及状态流转、跨系统约束、历史兼容、权限条件、异常兜底或非直观实现时，必须补充足够注释帮助后续维护。
- 注释要贴近具体代码，并在修改实现时同步更新；代码行为已变更时，旧注释必须一并修正或删除。
- `TODO`、`FIXME`、`NOTE` 这类标记允许保留英文标签，但内容说明使用中文，并写清触发条件或后续处理方向。

## 推荐模式

- 先观察目标文件和邻近模块的稳定写法，再以局部最小改动方式补充实现。
- 若目标文件历史上几乎没有注释，按“本次改动触达范围”渐进补齐，不要求为了补注释额外重写整文件，但也不要继续新增无语义代码。
- 文件级优先保证“看文件头就知道它负责什么”：类注释先交代职责、边界或所属业务，而不是只写空泛标题。
- 简单逻辑默认少注释，只在业务名词容易误解、字段含义不直观时补一行中文说明。
- 中等复杂逻辑在代码块前写 1 到 2 行中文注释，先概括本段处理目标，再进入分支或数据转换细节。
- 高复杂逻辑优先拆方法或拆组件；仍不可避免集中在一处时，按步骤写中文注释，让阅读者能快速理解处理顺序、关键判断和副作用。
- 对宿主约束、插件装载限制、外部接口兼容等“脱离上下文就难以看懂”的逻辑，优先在入口处补中文注释，而不是让维护者反向推理。
- 注释语言保持直接、可执行、可验证，尽量写成“这里是在处理什么约束”，不要写成空泛描述。
- 对实体字段、配置键、协议字段映射这类容易出现“值知道、语义不清”的位置，优先在字段或常量附近补中文注释，避免后续只能靠数据库表结构或外部文档反查。

## 禁止做法

- 为了“统一风格”大面积改写与当前需求无关的缩进、换行、import 顺序或命名。
- 因为邻近文件、历史模块或当前仓库普遍缺注释，就把“无注释”当作稳定风格继续沿用。
- 为了“看起来规范”给每一行代码都加注释，导致正文和注释互相干扰。
- 用注释复述显而易见的代码动作，例如“给变量赋值”“调用接口查询数据”。
- 在复杂逻辑已经发生变化后继续保留旧注释，导致代码和注释语义不一致。
- 大段堆砌背景说明，却不指出当前代码块真正处理的业务约束。
- 在注释中扩散敏感地址、凭据、token、内部环境信息或不应公开的联调细节。

## 开发或评审检查点

- 是否沿用了邻近代码的稳定风格，没有引入与任务无关的格式切换或大面积重排。
- 是否把“存量代码普遍缺注释”误判成当前应继续沿用的风格，而没有按本次变更范围补齐必要注释。
- 命名是否能直接表达业务含义，没有依赖模糊缩写或临时变量名撑完整段逻辑。
- 注释是否统一使用中文，并只在必要处保留英文术语或字段原文。
- 新增或修改的业务类、接口、配置类、Extract、Job 等文件，是否补了能说明职责边界的中文类注释。
- 新增或修改的关键方法、复杂代码块、配置常量、协议字段映射，是否补了足够的中文说明，而不是只留英文名或让维护者自行猜测。
- 注释是否解释了业务难点、边界条件和实现原因，而不是重复代码表象。
- 注释密度是否和业务复杂度匹配，既没有复杂逻辑缺注释，也没有简单逻辑过度注释。
- 修改代码后，相关注释是否同步更新，没有出现误导性旧说明。
