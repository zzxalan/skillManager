# 前端架构边界

## 适用场景

在 smart（空间管理）项目的 smart-core 基座 + smart-* 插件体系中处理前端页面、H5、微前端接入、接口请求、资源注册、构建链路或前端代码审查任务时，先读取本文件。

## 强制规则

- 先把项目识别为“宿主基座 + 插件”体系，不把插件前端误当成可独立部署的普通单页应用。
- 先确认真实前端入口、真实资源目录、真实构建链路和宿主消费方式，再决定改动点。
- 插件前端调用插件后端接口时，默认使用宿主暴露的 `/api/<插件ID>/...` 前缀，不直接写裸路径或自行猜测转发规则。
- 插件 Controller 的 `@RequestMapping` 只写插件内相对路径；前端不要因看到 Controller 局部路径而遗漏宿主挂载前缀。
- 页面应优先复用项目已有布局、请求封装、统一组件和邻近页面模式。
- 涉及页面导航、列表、表单、请求错误提示、H5 或微前端接入时，按对应 frontend reference 执行，不在页面内临时造一套并行方案。
- 涉及前端产物接入、资源复制、远程更新或接口联调时，同时检查后端打包脚本和插件装载约束。
- 涉及内网地址、token、账号密码、上传地址或示例凭据时，按敏感信息处理，不在新增文档、代码注释或交付说明中扩散。

## 推荐模式

- 先读当前插件的 `frontend/package.json`、`frontend/vite.config.ts`、页面目录、请求封装和相似页面，再决定实现方式。
- 先从 `backend/build.gradle` 或同类打包配置确认插件 ID，再组织前端 API 前缀。
- 调整路由、base、构建输出、H5 入口或微应用容器时，联动检查后端资源复制和最终访问路径。
- 若需要对接其他插件或宿主能力，优先寻找既有页面、services、api.ts 或统一封装中的真实用法。
- 在交付中区分“已在本仓库验证部分”和“依赖宿主联调部分”。

## 禁止做法

- 把插件前端直接当成普通 Vite 应用处理，并忽略宿主基路径、资源复制和插件装载。
- 脱离现有 React、Vite、qiankun、Ant Design、Pro Components、UnoCSS 与项目请求封装，引入并行页面体系。
- 只根据后端 Controller 的局部路径拼接前端 URL，忽略 `/api/<插件ID>/...` 真实访问形式。
- 修改入口、路由、base、输出目录或资源复制时只改一个文件，不联动检查相关配置。
- 把文档中的历史路径或历史类名直接当成当前事实。
- 在输出或新增文件中扩散敏感配置。

## 开发或评审检查点

- 前端入口、资源注册、构建输出和宿主消费路径是否以真实代码为准。
- 页面是否复用了现有布局、hooks、services、组件和邻近页面模式。
- 插件 API 路径是否包含正确插件 ID 前缀，且与后端打包配置一致。
- 涉及 PC/H5/微前端产物时，是否联动检查了构建输出、后端复制和最终访问路径。
- 是否把无法在当前仓库独立验证的部分说清楚。
- 是否避免在输出里扩散敏感配置。
