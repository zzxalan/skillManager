# 前端评审检查清单

## 适用场景

在提交前端实现、做前端代码审查、输出交付说明或检查前端规范符合度时，读取本文件。

## 核心检查

- 是否先遵守了 `architecture-boundary.md` 的宿主与插件边界。
- 是否保持了现有 React、Vite、qiankun、Ant Design、Pro Components、UnoCSS 和项目请求封装。
- 页面落点是否合理，是否沿用了现有目录、布局、hooks、services 与组件模式。
- 前端插件 API 路径是否包含正确的 `/api/<插件ID>/...` 前缀，且与后端打包配置中的插件 ID 一致。
- 前端列表或 CRUD 页面是否优先使用 `VAObkTable`。
- 页面导航与面包屑是否优先使用 `useBreadcrumb`。
- 标题、名称、描述、表格列值、卡片字段等文本区域是否考虑超长数据场景，并在必要时提供截断后的完整内容查看方式。
- 页面自行处理请求错误时，是否优先读取后端业务 `msg`，并避免请求层与页面层重复弹出错误提示。
- 页面级请求是否只由必要依赖触发，尤其是 `useEffectEvent` 返回函数没有被错误加入 `useEffect` 依赖数组。
- 若页面表现为“详情接口或事件接口持续重复请求”，是否先区分 `useEffectEvent` 依赖循环、开发环境 `StrictMode` 双调用，以及后端数据变更导致的真实参数变化。
- H5 页面是否使用独立入口、独立 Vite 配置、独立产物路径，并避免混入 qiankun。
- 涉及前端构建、H5、微前端入口或插件打包时，是否联动检查了产物接入链路。
- 是否沿用了邻近模块的稳定编码风格，没有做与任务无关的大面积格式重排。
- 代码注释是否统一使用中文，并按业务复杂度控制注释密度，重点解释业务约束而不是复述代码动作。
- 验证策略是否与改动范围匹配，是否优先采用最小必要验证，而不是为了形式追加无关的全量测试、全链路联调或重复构建。
- 是否说明了已验证部分与未验证部分。

## 审查优先级

1. 宿主挂载、构建产物和资源路径风险
2. 页面请求循环、鉴权路径和接口前缀风险
3. 统一组件、统一布局和统一请求封装是否被绕开
4. 文本溢出、错误提示、H5 适配等用户可见行为回归
5. 测试、构建和联调验证缺口

## 输出要求

- 先写发现的问题，再写简短总结。
- 优先写行为风险、兼容性风险、路径风险、用户可见问题和验证缺口，不只写命名或格式问题。
- 若仓库信息不足以支持结论，明确写出依据与不确定性。
