# Team Skill Iteration

## Goal

建立团队 skill 的反馈、记录、汇总评审和更新闭环。团队成员使用任意 team skill 过程中发现问题或改进机会时，先各自生成或追加迭代记录，评审时再汇总多人的记录，最后决定是否更新具体 skill。

## Workflow

1. 确认本次涉及哪个 skill、哪个任务场景，以及实际触发或执行过程。
2. 记录使用过程中遇到的问题或改进机会，包括触发不准、步骤缺失、规则过期、输出不可复用、执行踩坑、反复问题，或需要新增 `references/`、`scripts/`、`assets/`。
3. 区分“立即可修正文档错误”和“需要团队评审的规则变更”。
4. 将问题追加到本次使用者的迭代记录文件，默认状态为 `待评审`；记录格式以 `references/iteration-log.md` 为模板。
5. 给出建议落点，例如更新 `SKILL.md`、拆分 reference、补充脚本、调整 description、增加边界条件或补充验证方式。
6. 如用户明确要求执行更新，再修改对应 skill；否则只记录建议，不直接改动其他 skill。

## Resources

- 记录 skill 使用问题或改进建议时，读取 `references/iteration-log.md`，按模板追加到使用者自己的迭代记录文件。
- 团队定期评审、汇总多名成员记录或推进落地更新时，读取 `references/review-process.md`。

## Output

追加记录必须使用 `references/iteration-log.md` 中的分节模板，至少包含日期、记录人或来源、涉及 skill、触发任务或场景、遇到的问题、临时解决方式、建议优化、建议落点、优先级、状态、评审结论或关联变更。

交付时说明：

- 已追加到哪一个使用者记录文件。
- 本次建议属于文档错误修正，还是需要团队评审的规则变更。
- 后续若采纳，建议修改哪个 skill 文件或资源。

## Edge Cases

- 信息不足时，先用 `待补充` 标记缺失字段，并只询问会影响记录准确性的最小问题。
- 涉及生产、权限、密钥、客户数据或不可逆操作时，只记录风险和改进建议，不写入敏感值，并提示需要授权评审。
- 规则冲突时，记录冲突双方、实际采用的临时处理方式和需要评审的决策点。
- 用户只想记录不想修改时，只追加记录，不修改对应 skill。
- 用户明确要求立即修正文档笔误、路径错误或过期链接时，可以在记录后同步修正；涉及流程、规则、边界或默认行为变化时，必须先进入评审。
