# 团队 Skill 迭代评审流程

## 评审节奏

建议每 1-2 周收集团队成员各自生成的 skill 迭代记录，汇总后统一评审。如果出现 P0 或多个成员重复遇到同类问题，可以临时评审。

## 评审输入

- 各成员迭代记录文件中 `待评审` 状态的记录。
- 相关任务的实际输出、临时解决方式和验证结果。
- 被建议修改的 skill 文件、reference、script 或 asset。

## 汇总方式

1. 收集团队成员各自生成的迭代记录文件，不要求成员直接写同一个共享文件。
2. 将记录合并到本次评审汇总文件，保留“记录人或来源”和原始日期。
3. 按涉及 skill、问题类型和建议落点归类，合并明显重复的问题。
4. 对重复问题保留多个来源，避免只留下单条记录后丢失复现频次。
5. 汇总时不直接改具体 skill，只形成评审输入。

## 评审判断

1. 判断问题是否可复现，是否来自 skill 本身，而不是单次任务信息缺失。
2. 判断是否属于立即可修正文档错误，还是会改变团队规则、默认流程或边界条件。
3. 判断建议是否适合沉淀为通用规则；只适用于单个任务的偏好不进入 skill。
4. 判断落点是否合理：触发问题优先改 `description`，路由问题优先改 `SKILL.md`，细则优先改或新增 `references/`，重复机械步骤优先补 `scripts/`。
5. 决定优先级、负责人和状态：`已采纳`、`已拒绝` 或继续 `待评审`。

## 落地更新

采纳后再修改具体 skill，并保持单条规则只有一个权威落点：

- `SKILL.md`：只放目标、触发、路由、核心步骤和边界。
- `references/`：放详细规范、案例、检查清单和长流程。
- `scripts/`：放需要稳定复现的解析、转换、校验或生成逻辑。
- `assets/`：放模板、样例、静态资源。
- `agents/openai.yaml`：当展示名称、摘要或默认调用语发生变化时同步更新。

## 状态回写

更新落地后，在评审汇总文件中把记录状态改为 `已落地`。在“评审结论或关联变更”中写明修改文件、PR、commit 或评审结论；必要时再回写到成员各自的原始记录文件。

## 最小验证

每次修改 skill 后至少执行：

1. 检查 `name` 与目录名一致，`description` 是否覆盖触发场景。
2. 检查主 `SKILL.md` 是否仍然简洁，详细内容是否放入 `references/`。
3. 在本仓库运行 `python3 scripts/validate-skills.py`。
4. 用一条真实任务回放，确认 agent 能命中 skill 并读取正确资源。
