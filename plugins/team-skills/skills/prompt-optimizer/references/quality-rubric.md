# Prompt Quality Rubric

## 优化后的提示词结构

复杂任务优先使用这个结构：

```text
你是 <目标角色/执行对象>。

任务目标：
<一句话说明最终要完成什么>

背景与上下文：
<项目、业务、用户、数据、当前状态、已知事实>

需要参考：
<文件、目录、文档、接口、截图、数据源、命令>

执行范围：
<必须做什么；不要做什么；哪些文件或系统可动>

约束：
<代码风格、语言、性能、兼容性、安全、权限、时间、依赖>

工作步骤：
1. <先收集什么上下文>
2. <再如何分析或修改>
3. <最后如何验证>

输出格式：
<最终回复、文件、报告、表格、代码块或交付物格式>

验收标准：
- <可观察的完成条件>
- <测试、命令、检查项>

如果信息不足：
<哪些情况先提问；哪些情况可以基于假设继续>
```

## 项目上下文检查

优化 coding 或项目类提示词时，优先补全：

- 项目类型：前端、后端、全栈、移动端、脚本、文档、数据处理。
- 技术栈：语言、框架、包管理器、构建工具、测试工具。
- 本地约束：`AGENTS.md`、代码风格、命令前置条件、运行环境。
- 相关文件：入口文件、模块目录、测试目录、配置文件、已有相似实现。
- 期望变更范围：只分析、只生成提示词、实现代码、加测试、跑验证、写文档。
- 验证方式：单测、集成测试、lint、typecheck、截图、手动检查步骤。

## 缺口分类

把缺失信息分成三类处理：

- 可从项目上下文补全：直接写进提示词，并在补全依据中说明来源。
- 可合理假设：写成“默认假设”，不要装作事实。
- 必须确认：影响业务方向、数据安全、生产变更、成本、法律合规或用户体验的关键选择。

## 常见反模式

避免这些优化方式：

- 只把提示词写得更长，没有增加可执行信息。
- 加入与当前项目不一致的框架、命令或目录。
- 把用户没有决定的产品策略写成既定事实。
- 要求 agent 一次性完成多个互相冲突的目标。
- 缺少输出格式，导致结果不可比较。
- 缺少验收标准，导致完成状态不可判断。

## 简短任务模板

当用户只需要轻量优化时，使用更短的结构：

```text
请基于当前项目上下文完成以下任务：

目标：
<任务目标>

范围：
<相关文件/模块；不改动无关内容>

要求：
- <关键约束>
- <输出格式>
- <验证方式>

请先读取必要上下文，再给出结果；如果发现阻塞信息缺失，先提出需要确认的问题。
```
