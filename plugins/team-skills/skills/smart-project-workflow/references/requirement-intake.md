# 需求进入

## 适用场景

当需求只有口头描述、issue 描述不完整，或开发者准备让 AI 开始分析插件功能前，读取本文件。

## 必须澄清的问题

- 当前需求属于哪个插件，是否需要新增插件。
- 用户是谁，入口在哪里，核心业务动作是什么。
- 需求目标是什么，明确不做什么。
- 是否包含后端接口、前端页面、数据库、定时任务、事件、权限、菜单、路由、打包或部署。
- 是否依赖 smart-core 基座能力。
- 是否依赖其他插件的数据或能力。
- 是否涉及组织、校区、学院、部门、用户、角色、权限等基座主数据。
- 是否涉及状态流转、批量操作、删除、导入导出、异步任务或定时任务。
- 验收标准是什么，哪些场景必须验证。

## 输出要求

输出 `spec.md`，至少包含：

- 插件与功能名称。
- 背景与目标。
- 用户角色与入口。
- 功能范围。
- 不做范围。
- 业务规则。
- 依赖基座能力。
- 依赖其他插件能力。
- 验收标准。
- 未确认问题。

## 禁止做法

- 需求还没确认插件归属，就直接创建接口、页面或表。
- 把“做一个页面”“加一个接口”当成完整需求。
- 跳过不做范围，导致 AI 自动扩展需求。
- 把跨插件依赖当成本插件内部数据随意处理。
