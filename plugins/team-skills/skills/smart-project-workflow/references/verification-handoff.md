# 验证与交付

## 适用场景

当实现完成、准备交付、发起 PR 或让其他人 review 前，读取本文件。

## 验证层次

- 静态检查：格式、类型、lint、明显禁用模式。
- 后端验证：编译、单元测试、接口调用、分页、异常返回、数据写入分支。
- 前端验证：构建、页面渲染、路由、表格、表单、错误提示、超长文本。
- 插件接入验证：入口、资源、打包产物、基座装载、菜单路由、鉴权。
- 跨插件验证：`@Caller` / `@Supplier` 协议、非空入参、异常处理。
- 数据验证：迁移文件归属、表归属、首次插入、已有记录更新、状态流转。

## 强制规则

- 不能只写“已完成”，必须给出验证命令或验证证据。
- 不能把当前环境无法验证的内容写成已验证。
- 必须明确区分“已验证”“未验证”“需要联调环境验证”。
- 交付前必须按命中的 `smart-backend-standards/references/review-checklist.md` 或 `smart-frontend-standards/references/review-checklist.md` 自检；全栈或接入任务两者都检查。
- 涉及打包、资源、微前端或插件入口时，必须检查接入链路。

## 输出要求

输出 `verification.md`，至少包含：

- 验证环境。
- 执行命令。
- 验证结果。
- 已覆盖场景。
- 未覆盖场景。
- 需要基座联调的内容。
- 风险与回滚建议。

## PR 交接要求

PR 描述至少包含：

- 需求背景。
- 插件边界结论。
- 主要改动。
- API 和数据变化。
- 前端入口变化。
- 基座或其他插件依赖。
- 验证记录。
- 未验证部分。
- 是否更新 skill。

## Review 优先级

1. 是否破坏 smart-core 基座和插件装载。
2. 是否越权修改其他插件数据。
3. 是否绕过基座统一能力。
4. 是否遗漏打包、资源或路由接入。
5. 是否存在前后端契约不一致。
6. 是否存在关键验证缺口。
