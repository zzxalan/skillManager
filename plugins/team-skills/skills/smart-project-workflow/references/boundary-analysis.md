# 插件边界分析

## 适用场景

当任务涉及代码实现、数据读写、接口、前端页面、基座能力、其他插件能力、打包接入或联调时，读取本文件。

## 必须配合的实现层规范

边界分析必须触发当前团队中的实现层 standards：

- 后端、数据库、接口、调度、消息、打包联调：读取 `smart-backend-standards/references/architecture-boundary.md`
- 前端页面、路由、菜单、构建、微前端接入：读取 `smart-frontend-standards/references/architecture-boundary.md`
- 全栈、插件接入或跨端协作任务：两者都读取

如果任务后续进入实现，还要按任务类型继续读取对应 standards 的专题 reference。

## 强制判断

- 这个能力应该落在 smart-core 基座、当前插件，还是其他插件。
- 当前插件是否拥有相关数据表。
- 当前插件是否需要写表、改表或新增迁移文件。
- 是否存在其他插件已经提供的 `@Supplier`、公共查询接口、同步快照或只读视图。
- 是否需要通过 `@Caller` 调用其他插件能力。
- 是否需要注入 smart-core 基座 MAIN Bean。
- 是否涉及 `OrganizationService`、`TimeManager`、`EventBusManager`、Quartz、鉴权、菜单、路由、资源注册或插件装载。
- 是否存在无法在当前插件仓库独立验证的部分。

## 数据归属规则

- 默认只有所属插件可以维护自己的表结构和写入逻辑。
- 当前插件不能因为编译期可见或模型相似，就直接写入其他插件的数据表。
- 对其他插件数据的使用，优先通过跨插件能力、只读查询、同步快照或既有接口完成。
- 对基座组织、校区、学院、部门等主数据，优先走 smart-core 基座的组织服务能力。

## 输出要求

输出 `data-ownership.md`，至少包含：

- 当前插件职责。
- 基座职责。
- 其他插件职责。
- 数据表归属。
- 读写权限判断。
- 跨插件调用方式。
- 基座能力复用点。
- 风险与人工确认项。

## 禁止做法

- 把插件当成独立应用设计。
- 跳过 smart-core 基座能力，直接在插件内重造基础设施。
- 在非所属插件中新增迁移文件或修改表结构。
- 直接向其他插件拥有的数据表插入、更新、删除。
- 没有确认真实注入方式，就直接构造器注入基座 Bean。
