# 契约设计

## 适用场景

当需求涉及前后端协作、跨插件调用、数据读写、事件、定时任务、权限、菜单、路由、资源接入或外部系统对接时，读取本文件。

## 契约类型

- 后端接口契约：URL、方法、请求参数、响应结构、错误语义、分页结构。
- 前端页面契约：路由、菜单、面包屑、页面状态、列表列、筛选项、表单字段。
- 数据契约：表归属、字段语义、主键策略、唯一约束、状态流转、迁移文件归属。
- 跨插件契约：`@Caller` / `@Supplier`、非空入参、DTO、异常语义、兼容策略。
- 基座能力契约：MAIN Bean 注入、组织服务、时间能力、事件总线、Quartz、鉴权。
- 打包接入契约：前端产物路径、插件入口、资源注册、基座装载方式。

## 强制规则

- 前后端并行前，必须先定义接口契约。
- 跨插件调用前，必须先定义非空入参协议和返回语义。
- 数据库改动前，必须先确认表归属、迁移文件归属和回滚风险。
- 涉及基座能力时，必须先在现有插件中查找真实接入模式。
- 涉及菜单、路由、资源或微前端时，必须把打包接入纳入设计，而不是只设计页面代码。

## 输出要求

输出 `design.md`，至少包含：

- 功能方案概述。
- 插件边界结论。
- API 契约。
- 数据契约。
- 前端入口与页面结构。
- 跨插件或基座能力调用方式。
- 打包与接入影响。
- 风险点与降级方案。

## 推荐做法

- 对新增接口给出请求和响应示例。
- 对状态字段给出状态枚举和流转规则。
- 对前端列表、表单、详情页明确数据来源。
- 对跨插件调用明确调用方向和失败处理。
- 对无法本地验证的接入点提前写入验证计划。
