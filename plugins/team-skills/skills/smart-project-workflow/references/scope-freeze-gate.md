# 范围冻结闸门

## 适用场景

当任务来自 PRD、口头需求、用户要求“分配工作”、需要多个智能体协作、涉及 smart-core 与 smart-* 插件边界、或可能拆出需求/架构/UI/后端/前端/测试/发布/Grafana 等多类任务时，先读取本文件。

本文件用于防止在关键边界未确认前过早拆完整研发流水线。

## 冻结卡

创建实现类子任务前，必须能回答以下问题：

- 做什么：本阶段要交付的能力、行为或配置是什么。
- 不做什么：明确排除的页面、接口、导出、看板、接入、部署、数据迁移、兼容范围。
- 交付介质：代码 PR、技术方案、Grafana dashboard、数据库表、配置、测试报告、部署结果，还是只做调研。
- 数据归属：表、字段、统计口径、主数据、跨插件数据分别属于 smart-core、当前插件还是其他插件。
- 环境前置：目标仓库、分支、开发服务、数据库、Grafana、第三方服务、账号权限、PR 创建方式是否可用。
- 阶段验收：当前阶段验收到哪里；后续阶段哪些不纳入本轮。
- 依赖关系：哪些任务可并行探索，哪些必须等待前置产物。

## 推进规则

- 未冻结前，只允许创建需求澄清、代码调研、技术边界分析或最小验证任务。
- 需求、架构、UI/UX 可以并行探索，但实现、测试、部署、Grafana、审查默认必须等待前置产物满足。
- 子 issue 必须写清启动条件、依赖、交付物、不做范围和验证方式。
- 串行后续任务默认放 `backlog`，只有前置产物完成并被确认后再推进到 `todo`。
- 依赖真实环境的任务启动前必须确认环境存在；例如 Grafana 不存在时，不能启动 dashboard 配置任务。
- 实现任务默认要求 PR；若平台是 Gitea/GitHub/GitLab，应在任务描述中写明 PR 创建方式、分支命名和关联 issue key。

## 口径变更处理

用户补充的新口径覆盖旧范围时，协调者必须先做影响分析：

- 哪些 issue 继续有效。
- 哪些 issue 只保留为参考。
- 哪些 issue 需要取消。
- 哪些 issue 需要退回 `backlog`。
- 哪些 issue 描述、metadata、依赖和验收标准需要同步更新。

完成影响分析前，不继续推进旧口径下的实现、测试、部署或审查任务。

## 禁止做法

- 把 PRD 中出现的页面、导出、详情、看板等描述直接当成实现范围。
- 在未确认交付介质前同时启动后端、前端、Grafana、测试和审查。
- 环境未确认就启动部署、Grafana 或远程联调任务。
- 用“先做着看”替代范围冻结。
- 旧口径被用户覆盖后，让原子任务继续运行或继续触发后续任务。
