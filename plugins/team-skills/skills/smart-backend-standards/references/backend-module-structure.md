# 后端模块结构

## 适用场景

在新增后端包、Controller、Service、DTO、测试、插件入口相关代码时，读取本文件。

## 强制规则

- 优先沿用现有包结构和邻近模块组织方式，不为单个需求过度分层。
- 把业务逻辑放入 Service，把接口接入放入 Controller，把通用能力放入已有公共包或宿主能力。
- 验证范围要与改动范围和风险点匹配，优先做最小必要测试或构建验证，不为了交付形式补齐与本次改动无关的全量测试、全链路联调或重复构建。
- 新增测试放在 `backend/src/test/java`，只覆盖本次改动直接影响的关键装配、关键逻辑或关键资源声明。
- 涉及入口或资源注册时，联动检查真实入口类、资源目录与构建脚本。

## 推荐模式

- 先观察目标模块周围现有文件，按同样粒度补充类、DTO 和 Service。
- 在已有 `common`、`component`、`flex`、`listener`、`config` 等包内优先扩展，而不是平行再造一层基础设施。
- 先判断改动属于纯逻辑、接口接入、资源注册、构建链路还是宿主依赖，再选择对应的最小验证方式；能用单测、局部构建或静态检查覆盖的，不默认上升到全量构建或宿主联调。
- 若需求依赖宿主上下文，先写清边界，再做可在仓库内独立验证的部分。

## 禁止做法

- 只按个人偏好重排整个包结构。
- 把应复用的公共能力复制一份到新模块。
- 为了“看起来更完整”而追加与当前变更无关的全量测试、全链路联调或重复验证命令。
- 在缺少宿主环境验证的情况下宣称“完整可用”。

## 开发或评审检查点

- 新增类的位置是否符合现有包组织。
- 是否把 Controller、Service、公共基础能力混在一起。
- 验证是否覆盖了本次改动的直接风险点，既没有漏掉关键检查，也没有堆叠无关测试。
- 是否补了最基本的测试或说明未验证边界。
- 是否同步检查入口类、资源注册和构建链路。
